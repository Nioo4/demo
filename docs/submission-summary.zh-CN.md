# 项目简要说明

## 1. 项目定位

本项目是一个面向 ROOT 全栈岗位笔试题的 AI Native 应用生成 Demo。目标不是做一个完整的低代码平台，而是在有限时间内交付一条可运行、可演示、可扩展的核心闭环：用户输入需求，系统通过 AI 生成应用蓝图与代码片段，将结果以可视化方式展示，并支持持久化、回看、分享与基础项目管理。

---

## 2. 实现思路与关键取舍

### 2.1 总体实现思路

- 前端与后端统一使用 Next.js App Router，减少工程拆分成本，方便在同一项目内完成页面、API、鉴权与部署。
- 数据层使用 Supabase，承担用户认证、Postgres 持久化与项目隔离能力，保证 Demo 不只是前端假数据。
- AI 层同时保留 `Mock` 与 `Real AI` 两套生成模式：
  - `Mock` 用于稳定演示与本地开发。
  - `Real AI` 使用 DeepSeek，生成更真实的应用结果。
- 整体数据通过统一的 `GenerationProject` 结构贯穿，前端展示、API 存储、项目详情、分享页都复用同一套数据契约。

### 2.2 主流程设计

当前产品主流程为：

1. 用户在生成台输入需求，可附加图片/文档作为参考素材。
2. 后端调用生成服务，得到应用蓝图、Agent 阶段结果、代码片段与 artifacts。
3. 结果写入 Supabase，并在前端展示为：
   - 应用预览
   - Agent 时间线
   - VSCode 风格代码面板
   - artifacts 下载区
4. 用户可在项目库中继续进行搜索、筛选、排序、收藏、重命名、删除、分享等操作。

### 2.3 关键取舍

- 取舍一：没有把“生成结果”做成真正可执行的应用运行时，而是做成“蓝图预览 + 代码片段 + 项目产物”的组合展示。
  - 原因：在笔试时长内，优先保证完整闭环、稳定性与演示质量，而不是冒险做一个不稳定的动态执行沙盒。

- 取舍二：多 Agent 体验采用“阶段化展示”，而不是四个真正独立编排的 Agent 服务。
  - 原因：这样可以保留 AI 工作流的过程感，同时控制实现复杂度，避免把时间消耗在调度系统和复杂状态同步上。

- 取舍三：前端样式使用手写 CSS，而非引入 Tailwind 或额外 UI 框架。
  - 原因：项目规模不大，手写 CSS 更容易精确控制页面风格，也能减少依赖、降低部署风险。

- 取舍四：文件上传先支持“参考素材”能力，而不追求完整多模态解析。
  - 目前文本类文件可抽取内容摘要参与生成；复杂文档与图片更多用于辅助上下文。
  - 原因：先保证功能闭环，再逐步增强理解能力。

## 3. 当前完成程度

### 3.1 已完成

- 基础产品闭环
  - 需求输入、AI 生成、结果展示、项目保存、项目详情、公开分享均已打通。

- 真实数据持久化
  - 已接入 Supabase，项目、Agent 记录、artifacts 能够落库并在刷新后保留。

- 用户体系
  - 已接入 Supabase Auth，支持注册/登录。
  - 项目记录已按用户隔离，不再出现所有用户共享项目列表的问题。

- AI 能力
  - 已接入 DeepSeek，真实 AI 生成功能可用。
  - 同时保留 Mock 模式，便于稳定演示与调试。

- 前端体验
  - 已有中文生成台、项目库、项目详情页、公开分享页。
  - 代码展示升级为更接近 VSCode 的文件工作台样式。
  - 生成流程支持基础动效与状态反馈。

- 扩展功能
  - 支持 Prompt 模板库。
  - 支持上传图片/文档作为参考素材。
  - 支持 artifacts 保存与下载。
  - 支持项目搜索、筛选、按时间排序、重命名、删除、收藏。
  - 支持公开/私有切换与分享链接复制。

- 工程交付
  - 已具备本地运行、类型检查、数据库测试、真实 AI 测试与 Vercel 部署能力。

### 3.2 当前仍未完成或完成度不足的部分

- 生成结果预览仍偏“蓝图展示”，不是一个真正可执行、可交互的应用运行页。

- 当前的多 Agent 更接近“单次生成结果的阶段化呈现”，还不是真正的多阶段独立编排与协作。

- 上传文件的理解能力仍有限：
  - 文本类文件支持较好；
  - 图片、PDF、DOCX 等复杂内容还没有做到高质量解析、OCR 或结构化提取。

- 注册登录流程尚未做生产级收口：
  - 例如自定义 SMTP、邮件验证策略、异常提示与限流体验还可以继续完善。

- 自动化测试还不够完整：
  - 当前已有类型检查与部分真实链路脚本；
  - 但还缺完整的 E2E 回归测试与更系统的异常场景覆盖。

## 4. 如果继续投入时间，我会如何扩展

### P0：优先补强“结果可用性”和“完成度”

- 做受控的预览渲染 schema
  - 让 AI 输出更细粒度的页面结构，由前端渲染成更像真实产品的预览，而不是只展示蓝图信息。
  - 原因：这项改动最能同时提升完成度、用户体验和展示说服力。

- 完善异常处理与测试回归
  - 补齐关键链路的错误提示、失败兜底、加载状态与回归测试。
  - 原因：这是提升稳定性和可交付性的最直接投入。

### P1：优先补强“工程思维”和“扩展潜力”

- 把多 Agent 做成真正分阶段执行
  - 例如 `planner -> ux -> coder -> qa` 串行生成，并把每阶段结果独立存成 artifacts。
  - 原因：这会让系统架构更有层次，也更符合 AI 工作流产品的定位。

- 强化 artifacts 体系
  - 增加更多结构化产物类型，例如页面 schema、接口草案、测试清单、交互说明等。
  - 原因：比单纯增加页面更能体现“产物链”思维。

### P2：优先补强“产品化能力”

- 接入更完整的文件理解能力
  - 包括 PDF/DOCX 正文抽取、图片 OCR、图像理解等。

- 完善用户体系
  - 加入自定义 SMTP、密码重置、邮箱验证策略、账号信息页等。

- 增加导出与协作能力
  - 如导出 Markdown/JSON、生成分享摘要、多人协作查看等。

### 优先级判断依据

- 优先做能直接提升题目评分项的能力：完成度、用户体验、可交付性。
- 其次再补工程深度：真实多 Agent、结构化产物链。
- 最后再做更偏长期产品化的能力：高级文件理解、协作与导出。

---

## 5. 总结

当前版本已经完成了从需求输入到 AI 生成、结果展示、项目持久化、用户隔离、分享与项目管理的完整闭环，具备较强的演示性与可交付性。后续如果继续迭代，最值得投入的方向不是继续堆叠零散功能，而是把“生成结果”做得更真实可用，并进一步增强多 Agent 与 artifacts 的工程深度。
